# 全体仕様

## 1. プロジェクト概要

本プロジェクトは、traQ（最大ユーザー数：約700名、チャンネル数：約7,000）上のリアルタイムなコミュニケーション・アクティビティ（投稿・閲覧）を、直感的かつ美しい「光の島と波紋」として可視化するWebアプリケーションである。
ユーザーに「今、traQのどこが盛り上がっているか」「人々の関心がどう動いているか」を視覚的に伝え、システム全体の活気を表現することを目的とする。

## 2. ターゲット規模と前提条件

* **対象環境**: traQのライブ環境
* **最大チャンネル数**: 約 7,000 チャンネル
* **最大アクティブユーザー（同時接続）**: 約 700 ユーザー
* **バックエンドインフラ制約**: メモリ使用量 150MB 以下
* **利用技術スタック**:
* フロントエンド: Vue / WebGPU
* バックエンド: Go
* 通信プロトコル: SSE (Server-Sent Events)



## 3. コア体験（UI/UXとビジュアルアイデンティティ）

画面全体を暗い宇宙のような空間とし、チャンネル群を「光を放つ星（ノード）」として表現する。

### 3.1 チャンネルツリーと島の構成

* **島の形成**: ルートチャンネルごとに「島」を形成し、それぞれに固有の色（全9色）を割り当てる。
* **ツリー配置**: ルートを中心とし、子チャンネルは外側へ広がるように配置する。外側（子）に行くほどノードの色や明るさを減衰させる。

### 3.2 アクティビティの視覚化（状態と減衰）

* **盛り上がり度**: 各チャンネルの熱量は、ノードの「サイズ」と「明るさ（発光）」で表現する。
* **時間減衰**: 盛り上がったチャンネルは、時間経過とともに自然かつ滑らかに小さく、暗くなっていく（フロントエンドの毎フレーム描画による自律的な減衰）。
* **フィーバー状態**: 一定以上の盛り上がり度に達したチャンネルは、輪郭の強調や強い発光などの特殊なエフェクトを付与する。

### 3.3 インタラクション（光のイベント表現）

ユーザーのアクションは、対照的な2つの光のアニメーションとして表現する。

1. **投稿（MessageCreated）**
* **表現方針**: ツリー構造を利用した「エネルギーの注入と熱の拡散（波紋）」
* **視覚効果**: ルートチャンネルから対象の子チャンネルへ向けて光の粒子が飛び、着弾した瞬間に波紋（Ripple）が発生する。
* **熱の伝播**: 波紋が親チャンネルに到達するごとに親が発熱し、親からも波紋が連鎖的に発生する。これをルートに到達するまで繰り返す。


2. **閲覧（ChannelWatched / 移動）**
* **表現方針**: 個人の回遊を示す「細い光の線（ビーム）」
* **視覚効果**: 直前まで閲覧していたチャンネルから、新しく閲覧し始めたチャンネルへ向けて、細く直進的な光のビームが飛ぶ。投稿よりも主張を抑え、背景のノイズとしてシステムの鼓動を演出する。



## 4. 全体アーキテクチャと責務の分割

パフォーマンスと表現の自由度を両立するため、システムは「イベントの集約（バックエンド）」と「アニメーション・状態のシミュレーション（フロントエンド）」に完全に分離する。

### 4.1 バックエンド

* **データ収集と推論**: traQ APIと接続し、リアルタイムな投稿イベントや、定期的な観測に基づくユーザーの閲覧移動履歴を収集・推論する。
* **最小限のインパルス送信**: イベント発生時、具体的なスコア計算や親への伝播計算は行わず、「どこで投稿/移動があったか」という軽量なトリガー（インパルス）のみを即時配信する。
* **定期的な状態の同期**: 一定時間ごとにサーバー側で正しい「盛り上がり度」を計算し、状態に変化があったチャンネルのリストのみをフロントエンドへ定期配信する。

### 4.2 フロントエンド

* **自律的な描画ループ**: バックエンドからの通信に依存せず、独自のレンダリングループ（60fps）を回し、常に全ノードのサイズや不透明度を時間減衰させる。
* **イベントの視覚化と連鎖**: バックエンドから受信した軽量なトリガーを元に、パーティクル、波紋、ビームの描画を開始する。
* **状態計算の委譲**: 波紋が親ノードに衝突した際の「盛り上がり度の加算（伝播）」など、視覚と密結合した状態変更はすべてフロントエンド内で完結させる。定期的に届くバックエンドからの「答え合わせ」データを用いて、内部状態を滑らかに補正する。

### 4.3 通信方式

リアルタイムかつ単方向のデータストリームに最適なSSEを採用する。

* メモリ保護のため、巨大な初期化データ（7,000チャンネルの構造）はサーバーメモリ上にJSON文字列として単一キャッシュし、新規接続時にそのまま流し込む。
* 差分データとイベントトリガーのみをブロードキャストし、ネットワーク帯域とサーバーリソースを最小化する。

## 5. 非機能要件

* **パフォーマンス**: 700人が同時接続し、活発な投稿が行われている状態でも、バックエンドのメモリが150MBを下回ること。フロントエンドは一般的なPCブラウザで60fpsを維持できること。
* **スケーラビリティ**: チャンネル数や閲覧ユーザー数が微増した場合でも、SSEのペイロードサイズが急増しない設計（差分更新・イベント駆動）を維持すること。
