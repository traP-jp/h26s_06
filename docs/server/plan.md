### 🛠️ フェーズ1: プロジェクト基盤の構築と外部連携

まずは外部システム（traQ）からリアルタイムにデータを受け取り、メモリ上に静的なツリー構造を作り上げる土台のフェーズです。

* **1.1 プロジェクトの初期化と環境設定**
* `go mod init` によるモジュール作成と、環境変数（traQのAPIトークン、ポート番号など）のロード機構の実装。


* **1.2 traQ APIクライアントの構築**
* 起動時にtraQ APIを叩き、全チャンネル情報（約7000件）を取得する処理の実装。
* WebSocketに接続し、`MessageCreated`（投稿）などのリアルタイムイベントを受信するリスナーの構築。


* **1.3 Grand Rootツリーの初期構築とJSONキャッシュ**
* 取得したチャンネル情報を `GrandRootID` を頂点とする単一のフラットマップ構造にマッピング。
* 構築完了後、フロントエンドへ渡すための `init` ペイロードを一度だけJSONシリアライズし、メモリ上に `[]byte` としてキャッシュする処理の実装。



### 🧠 フェーズ2: インメモリ状態管理と排他制御

システムの中核となるデータ構造と、ゴルーチン間の競合を防ぐスレッドセーフな状態管理マネージャーを実装します。

* **2.1 `StateManager` の実装**
* `channels` マップと `users` マップ、および全体を保護する `sync.RWMutex` の組み込み。


* **2.2 ユーザー状態と「移動検知」ロジックの構築**
* traQからのイベントを解析し、ユーザーの現在の閲覧チャンネルとメモリ上の `CurrentChannel` を比較する処理の実装。
* 変化があった場合にのみ、「どこからどこへ移動したか（`from`, `to`）」の差分データを抽出して内部キューに送るロジックの完成。


* **2.3 ボット除外フィルターの適用**
* 投稿イベントがボットによるものか判定し、不要な処理をスキップする軽量なフィルタリングの実装。

* **2.4 監視対象の運用**
* 認証した人が閲覧してるチャンネルと閲覧している人を取得する
* 上で監視していないチャンネルのうち、盛り上がり度の上位を確率が高いように、いくつかを監視対象にする
* 前回監視対象に入ってから時間が経てば経つほど、監視対象に入れる確率を高くする

### 🚀 フェーズ3: SSE配信エンジンとリソース保護

構築したインメモリ状態を、最大700人のクライアントへリアルタイムかつ安全にストリーミングするフェーズです。

* **3.1 セマフォ制御付きSSEハンドラーの実装**
* `text/event-stream` のエンドポイント作成。
* 150MBのメモリスパイクを防ぐため、`maxConcurrentInits`（例: 10）を設定したチャネルによるセマフォ制御を導入し、キャッシュ済み `init` JSONを安全に書き込む処理の実装。


* **3.2 クライアント接続プールとブロードキャスターの実装**
* 接続中の全クライアントのGoチャネルを管理するプールの作成。
* 切断検知時にリソースを安全に解放するリーク防止機構の実装。


* **3.3 インパルス（`trigger`）配信の統合**
* フェーズ2で構築した「投稿」と「移動」のイベント発生時に、最小限のペイロード（`msg`, `mov`）を即座にシリアライズして全クライアントへブロードキャストする処理の連携。



### ⚙️ フェーズ4: 確率的同期ロジックとパフォーマンス最適化

システムの整合性を保つためのスコア同期アルゴリズムと、非機能要件（150MB・60fpsの支援）を満たすための最終調整フェーズです。

* **4.1 確率的同期タスク（Ticker）の実装**
* 30秒ごとに動作する `time.Ticker` ループの作成。
* 全チャンネルを走査し、スコア変化量（$\Delta S$）と経過時間（$\Delta T$）を変数とする確率 $P$ を算出する計算式の実装。
* 乱数判定をクリアしたチャンネルのみを抽出し、`sync` ペイロードとして配信するロジックの完成。


* **4.2 スコア加算・自律減衰ロジックの仮実装**
* バックエンド側で正解となるスコアを加算・時間減衰させる数式を組み込み、同期データに「動き」を持たせる（※具体的な数式は別途定義）。


* **4.3 メモリプロファイリング（pprof）と限界負荷試験**
* `net/http/pprof` を組み込み、ダミーの700クライアントを接続させ、秒間数十件のイベントを強制発生させる高負荷テストの実施。
* ヒープメモリの消費が150MB以内に収まっていることを確認し、必要に応じてGCパラメータやバッファサイズの微調整を行う。
