# フロントエンド実装における重要な考慮事項

アーキテクチャ設計を実運用レベルに引き上げるため、実装時に直面しやすい課題と対策を整理する。

## 1. タブのバックグラウンド遷移時の状態不整合

### 課題

ブラウザのタブが裏に回ると、仕様により `requestAnimationFrame` の描画ループは一時停止する。一方で、バックエンドからの SSE 通信は裏側でも受信され続ける。

その結果、ロジック層のスコアだけが無限に加算され、時間経過による自律減衰が止まる。タブに戻った瞬間に、画面全体が真っ白に発光してフリーズする、あるいは溜まっていた波紋アニメーションが一気に爆発するといった不具合が発生する。

### 対策

`Page Visibility API` の `document.visibilityState` を監視し、以下を制御する。

- 非アクティブ時
  - アニメーションを止める。
  - 波紋やビームの新規生成をスキップする。
  - SSE のインパルスは受け付ける。
  - 時間経過による減衰は `setInterval` などで簡易的に計算し続ける。
- 復帰時
  - アニメーションキューをリセットする。
  - 直後に来るサーバーからの `activity_update` に対しては、`Lerp` ではなく `Snap` で即時上書きする。

## 2. 3D 空間のクリック・ホバー判定

### 課題

特定のチャンネル情報を Vue 側に表示するには、3D 空間上の 7,000 個の星の中からマウスカーソルと交差するものを探すピッキングが必要になる。

`InstancedMesh` に対する素朴な Raycasting を毎フレーム、または `mousemove` ごとに実行すると、CPU 負荷が高まり FPS が低下する。

### 対策

- スロットリング
  - `mousemove` イベントでの Raycast 処理を間引く。
  - 例えば 100ms に 1 回程度に制限する。
- BVH の導入
  - Three.js の標準 `Raycaster` は全ノードに対して計算を行う。
  - 空間分割ツリーを用いて当たり判定を高速化する `three-mesh-bvh` の導入を検討する。
  - 初期化時の計算コストは増えるが、実行時の負荷は大きく下がる。

## 3. Retina ディスプレイと Bloom パスの罠

### 課題

`EffectComposer` と `UnrealBloomPass` は、画面のピクセルごとに重いブラー処理を行う。

MacBook の Retina ディスプレイや最新のスマートフォンのように `Device Pixel Ratio` が 2 や 3 の環境でレンダラーをフル解像度のまま動かすと、描画ピクセル数が 4 から 9 倍になり、一気に重くなる。

### 対策

WebGL レンダラーのピクセル比に上限を設ける。Bloom の滲みが少し粗くても見た目は十分に保てるため、パフォーマンスを優先する。

```js
// DPR が 3 の端末でも、最大 1.5 に抑える
renderer.setPixelRatio(Math.min(window.devicePixelRatio, 1.5));
```

## 4. 動的エフェクトのメモリリークと GC 対策

### 課題

波紋やビームを、イベントが来るたびに `new THREE.Mesh(...)` で生成し、終わったら `scene.remove(...)` で消去していると、ガベージコレクションが頻発する。結果として、画面が定期的に引っかかるマイクロスタッターの原因になる。

### 対策

オブジェクトプールを使う。

- 波紋用とビーム用のメッシュをあらかじめ一定数生成しておく。
- 配列としてプールに保持する。
- イベント発生時は、プールから非表示中のメッシュを取り出す。
- 座標と時間をリセットして表示状態にする。
- アニメーション終了後は再び非表示にしてプールへ戻す。

描画ループ中に `new` と `dispose` を行わない設計にする。

## 5. WebGL コンテキストの消失

### 課題

PC がスリープから復帰した時や、GPU メモリが不足した時に、ブラウザが意図的に WebGL コンテキストを破棄することがある。この場合、画面は真っ黒になる。

### 対策

`canvas` 要素で `webglcontextlost` イベントを監視する。発生した場合は無理に復旧させず、Vue 側で「描画エラーが発生しました。リロードしてください」と案内するフォールバック UI を出す。

## 6. 初回計算時のメインスレッドブロック

### 課題

アプリ起動時には、7,000 ノードの黄金角・再帰的な 3D 座標計算と Three.js のシェーダーコンパイルが同時に走る。この間、JavaScript のメインスレッドが占有され、ブラウザ全体が数秒間フリーズしたように見える。Vue のローディングスピナーすら止まる。

### 対策

- トポロジー計算の非同期化
  - 座標計算アルゴリズムを Web Worker に分離する。
  - もしくは数フレームに分けて計算する。
  - `generator` 関数や `requestIdleCallback` を使う。
- プレコンパイル
  - `renderer.compile(scene, camera)` をローディング画面の裏で明示的に呼ぶ。
  - シェーダーのコンパイル完了後に画面をフェードインさせる。

## 7. SSE 通信の一時切断と再接続戦略

### 課題

ユーザーがスマホで地下鉄に乗るなどしてネットワークが瞬断すると、SSE `EventSource` が切断される。デフォルトの `EventSource` は自動再接続を試みるが、切断中に発生したインパルスは失われ、フロントエンドとバックエンドの状態がずれる。

### 対策

- `EventSource` の `onerror` をフックする。
- 指数的バックオフで再接続を管理するカスタムロジックを組み込む。
- 再接続成功時は通信を再開するだけでなく、サーバーに現在の全チャンネルの正確なスコアのフルスナップショットを要求する。
- あるいは次の `activity_update` を即時適用して、切断中のズレをリセットする。

## 8. UI とカメラ操作のイベント競合

### 課題

画面の端に Vue で実装したチャンネル詳細パネルや設定ボタンを配置するとする。ユーザーがパネル上のテキストをドラッグで選択したりスクロールしたりした時、背後の Three.js の `OrbitControls` まで反応し、意図せず宇宙空間が回転・ズームしてしまう。

### 対策

- Vue 側の UI コンポーネント最上位要素でイベント伝播を止める。
- `@pointerdown.stop` や `@wheel.stop` を徹底する。
- あるいは CSS で UI のコンテナに `pointer-events: auto;` を設定し、それ以外を `pointer-events: none;` にして、キャンバスと操作領域を分離する。

## 9. ブラウザのリサイズへの追従

### 課題

PC ブラウザでウィンドウサイズを変えたり、スマホで画面を回転させたりした時に、Three.js のキャンバスサイズとカメラのアスペクト比が更新されないことがある。その結果、星が楕円形に潰れたり、画面の一部が切れたりする。

### 対策

`onMounted` で `window.addEventListener('resize', handleResize)` を登録し、リサイズ時に必ず次の 3 つをセットで更新する。

```js
camera.aspect = window.innerWidth / window.innerHeight;
camera.updateProjectionMatrix();
renderer.setSize(window.innerWidth, window.innerHeight);
```
